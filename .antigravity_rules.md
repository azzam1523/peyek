# Project Rules & Research: Action Fish Shooter

## 1. Security & Architecture Analysis (Inspired by Jackpot Fishing)

*   **Engine**: Original game uses **Cocos Creator v2.4.6** with **WebGL/Canvas renderer**. This means all elements (fish, bullets, balance) are drawn directly on canvas, making them hard to manipulate via DOM.
*   **Server-Authoritative State**: Client balance and fish HP are only visual. The server must be the source of truth.
- **Server-Side Cooldowns**: Enforce a minimum of 200ms between shot requests on the server.
- **Server-Side Damage**: The server calculates damage based on weapon and bet, client only sends a "hit" intent.

## 2. Feature Implementation Notes

### Auto Fishing (V2)
- **Behavior**: 
  - If OFF: Opens a selection panel to choose target fish.
  - If ON: Turns OFF immediately with a single click (no popup).
- **Targeting**: Filters current fish based on `selectedAutoTargets`. If empty, targets all fish.
- **Smart Logic**: Automatically picks the highest-value fish from the allowed selection.

### Frozen Skill
- **Visuals**: Pulsing ice-blue overlay (0.35 alpha) with a Sine ease.

### Target Mode
- **Visuals**: Continuous laser beam with rotating crosshair.
- **Ammo**: Fires invisible piercing bullets to maintain "laser" feel while keeping discrete coin deduction.

## 3. iTech Labs Baseline (RTP & Fairness) - Internal Notes

- iTech Labs does **not** publish one universal fish-value table to copy.
- For compliance direction, use:
- RNG fairness and unpredictability as core.
- Mathematical RTP model that is reproducible from config (spawn rate, HP, payout, bonus logic).
- Practical RTP monitoring in simulation/live logs so actual RTP tracks theoretical RTP within acceptable variance.
- Change control:
- Any HP/score/probability rebalance must be documented with version/date and expected RTP impact.
- Keep bonus triggers (jackpot/super prize/golden events) auditable and bounded.
- Implementation policy in this project:
- Fish-value balancing uses linear-risk progression to keep volatility and RTP easier to audit.
- Avoid hidden payout multipliers outside documented logic.
- If future compliance target is requested (example: 96.00% RTP), run dedicated Monte Carlo simulation and store report snapshot.
